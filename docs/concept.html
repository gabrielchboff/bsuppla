1. Introduction

bsuppla is built on a foundation of well-established software engineering principles, combined with security-specific design patterns. This document explains the technical concepts that make bsuppla extensible, testable, and safe.

2. Design Principles

The architecture follows several key design principles:

2.1 Open/Closed Principle

Software entities should be open for extension but closed for modification. In bsuppla, this means:

  • Adding new detectors requires no changes to existing code
  • Each detector is isolated in its own module
  • New detection logic is added by implementing the Detector trait

2.2 Single Responsibility

Each component has a single, well-defined responsibility:

Component Responsibility
detector.rs Core trait definitions and types
scanner.rs Filesystem traversal and orchestration
detectors/* Specific detection logic implementations

2.3 Dependency Inversion

High-level modules do not depend on low-level modules. Both depend on abstractions. The scanner depends on the Detector trait, not concrete implementations.

3. The Detector Trait

The Detector trait is the foundation of bsuppla's extensibility:

pub trait Detector: Send + Sync {
    fn name(&self) -> &'static str;
    fn detect(&self, ctx: &FileContext) -> Option<Finding>;
    fn severity(&self) -> Severity {
        Severity::Medium
    }
}

3.1 Trait Requirements

  • Send + Sync: Ensures thread-safe usage in the detector registry
  • name(): Returns a static string identifier for the detector
  • detect(): Performs the actual detection logic, returning an optional Finding
  • severity(): Default implementation returns Medium severity

3.2 Implementing a Detector

Here's an example of implementing a custom detector:

use crate::detector::{Detector, FileContext, Finding, Severity};

pub struct CustomDetector;

impl Detector for CustomDetector {
    fn name(&self) -> &'static str {
        "custom_detection"
    }

    fn detect(&self, ctx: &FileContext) -> Option<Finding> {
        // Custom detection logic here
        if ctx.is_executable && /* ... */ {
            Some(Finding::new(
                "custom_detection",
                ctx.relative_path.clone(),
                "details".to_string(),
                Severity::High,
            ))
        } else {
            None
        }
    }
}

4. FileContext Structure

The FileContext provides detectors with pre-computed file metadata:

pub struct FileContext<'a> {
    pub path: &'a PathBuf,           // Absolute path
    pub relative_path: &'a PathBuf,   // Relative to scan root
    pub mode: u32,                     // Unix permissions
    pub is_executable: bool,
    pub is_world_writable: bool,
    pub is_world_readable: bool,
    pub is_suid: bool,
    pub is_sgid: bool,
}

4.1 Why Pre-computed Metadata?

Pre-computing metadata in the scanner provides several benefits:

  • Performance: Metadata is collected once per file
  • Consistency: All detectors see the same data
  • Simplicity: Detectors don't need to perform filesystem operations

5. DetectorRegistry

The DetectorRegistry manages all registered detectors and provides the main detection interface:

pub struct DetectorRegistry {
    detectors: Vec<Box<dyn Detector>>,
}

impl DetectorRegistry {
    pub fn new() -> Self;
    pub fn register(&mut self, detector: Box<dyn Detector>);
    pub fn detect(&self, ctx: &FileContext) -> Vec<Finding>;
}

5.1 Registration Pattern

Detectors are registered in the default_registry() function:

pub fn default_registry() -> DetectorRegistry {
    let mut registry = DetectorRegistry::new();
    registry.register(Box::new(permissions::SuidSgidDetector));
    registry.register(Box::new(credentials::PrivateKeyDetector));
    // ... more detectors
    registry
}

5.2 Detection Process

When detect() is called:

  1. Iterates through all registered detectors
  2. Calls each detector's detect() method
  3. Collects all non-None results into a vector
pub fn detect(&self, ctx: &FileContext) -> Vec<Finding> {
    self.detectors
        .iter()
        .filter_map(|d| d.detect(ctx))
        .collect()
}

6. Severity Scoring

The severity system provides a consistent way to assess the risk level of findings:

pub enum Severity {
    Critical,  // Score: 5
    High,       // Score: 4
    Medium,     // Score: 3
    Low,        // Score: 2
}

impl Severity {
    pub fn score(self) -> u32 {
        match self {
            Severity::Critical => 5,
            Severity::High => 4,
            Severity::Medium => 3,
            Severity::Low => 2,
        }
    }
}

6.1 Suspicion Level Calculation

The overall suspicion level is computed by summing individual signal scores:

fn suspicion_level(findings: &[&Finding]) -> &'static str {
    let mut score = 0u32;
    for f in findings {
        score += f.severity.score();
    }
    
    if score >= 10 { "high" }
    else if score >= 4 { "medium" }
    else { "low" }
}

0-3 Low 0
4-9 Medium 1
10+ High 2

8. Extensibility

Adding new detection capabilities is straightforward:

8.1 Create a New Detector

  1. Create a new file in src/detectors/
  2. Implement the Detector trait
  3. Add tests for the detector

8.2 Register the Detector

  1. Add the module to detectors/mod.rs
  2. Register in default_registry()

8.3 Example: Adding a New Threat Detector

// src/detectors/new_threat.rs

use crate::detector::{Detector, FileContext, Finding, Severity};

pub struct NewThreatDetector;

impl Detector for NewThreatDetector {
    fn name(&self) -> &'static str {
        "new_threat_detection"
    }

    fn detect(&self, ctx: &FileContext) -> Option<Finding> {
        // Detection logic
        if /* threat condition */ {
            Some(Finding::new(
                "new_threat_detection",
                ctx.relative_path.clone(),
                "description".to_string(),
                Severity::High,
            ))
        } else {
            None
        }
    }
}

9. Safety Guarantees

bsuppla is designed with safety as a primary concern:

9.1 Zero Execution

  • Files are never opened with execute permissions
  • No code from the image is ever run
  • Analysis is strictly read-only

9.2 Memory Safety

  • Written in Rust - memory safe by default
  • No raw pointers or unsafe blocks in detection logic
  • All file operations use safe abstractions

9.3 Isolation

  • Each detector is isolated - one cannot affect another
  • Panics in detectors are contained
  • Analysis runs in a controlled order

9.4 No Network Access

  • Analysis phase requires no network connectivity
  • Only Docker operations (pull/save) require network
  • Can be run in air-gapped environments

10. Conclusion

bsuppla demonstrates how sound software engineering principles can be applied to security tooling. The combination of:

  • Trait-based plugin architecture
  • Clear separation of concerns
  • Comprehensive test coverage
  • Safety-first design

Results in a tool that is both extensible and trustworthy.
