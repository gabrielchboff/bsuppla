Architecture

Understanding how bsuppla works under the hood.

Goal

Statically analyze Docker images without executing them, identifying suspicious binaries that may indicate supply-chain attacks.

The primary objectives are:

  • Provide visibility into container image contents without runtime risk
  • Detect common indicators of compromise in container images
  • Enable automated security scanning in CI/CD pipelines
  • Support allowlisting and baseline comparisons for operational flexibility

Scope

Category Included Excluded
Input Docker image as .tar Running containers
Analysis Static file inspection Runtime behavior
Detection Heuristic patterns Malware signatures
Intelligence Built-in heuristics Machine learning

Components

1. Image Loader

Reads the Docker image tar archive and parses the manifest to identify individual layers. Handles both v1 and v2 image formats.

  • Parses manifest.json
  • Identifies layer.tar files
  • Extracts image configuration

2. Layer Extractor

Extracts each layer's layer.tar and reconstructs the complete filesystem by layering them in order. Handles file permission preservation and symbolic links.

  • Extracts layer contents
  • Handles layer ordering
  • Preserves permissions/ownership
  • Processes whiteouts

3. Binary Analyzer

Scans the extracted filesystem for various security indicators. This is the core detection engine.

  • ELF file detection and analysis
  • File permission checks
  • Entropy calculation for packed binaries
  • Pattern matching for known threats

4. Signal Detector

Matches findings against known threat patterns and categorizes them by severity.

  • Crypto miner detection
  • Private key identification
  • Suspicious tool detection
  • Credential file scanning

5. Report Generator

Compiles findings into a human-readable report with suspicion levels and optional baseline comparison.

Pipeline Details

Stage 1: Image Acquisition

bsuppla uses Docker to pull and save the image, or can work with an existing tar file. This stage is skippable via BSUPPLA_SKIP_DOCKER.

Stage 2: Extraction

Each layer is extracted and the filesystem is reconstructed. Files in later layers override earlier ones. Whiteout files mark deleted files.

Stage 3: Analysis

The analyzer walks the filesystem and applies various checks to each file. Key checks include:

  • Is the file an ELF binary?
  • Is it stripped, static, or packed?
  • Are permissions suspicious (SUID, world-writable)?
  • Is it in an unusual location?

Stage 4: Detection

Findings are matched against known threat patterns. Each match produces a signal with type, path, and context.

Stage 5: Reporting

Final report is generated, applying any allowlist entries and comparing against baseline if provided.

Trust Model

  • Docker image is untrusted input
  • Tool must never execute image contents
  • Analysis is read-only
  • No network access during analysis
Security

bsuppla performs static inspection only. It never executes any files from the analyzed image, making it safe to scan potentially malicious images. However, always run in an isolated environment as a defense-in-depth measure.

Threats Addressed

bsuppla is designed to detect the following threat categories:

Supply-Chain Attacks

  • Backdoored base images
  • Compromised dependencies embedded in images
  • Trojaned binaries

Obfuscation

  • Packed or encrypted binaries
  • Stripped executables
  • Static binaries that bypass dynamic analysis

Covert Execution

  • Hidden executables in dotfiles
  • Executables in unusual locations
  • Startup scripts in init/rc/cron

Credential Theft

  • Exposed private keys
  • SSH authorized keys
  • Credential files (.env, .aws, .docker)
  • Insecure /etc/shadow

Cryptocurrency Mining

  • Known miner binaries
  • Mining pool configurations
