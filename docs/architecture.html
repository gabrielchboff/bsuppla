System Architecture

This document provides a detailed architectural description of the bsuppla static analysis framework.

Table of Contents

  • Design Goals
  • Component Overview
  • Detector Framework
  • Processing Pipeline
  • Trust Model
  • Extensibility

1. Design Goals

The architecture is driven by the following design requirements:

Goal Description Priority
Zero Execution Never execute any code from analyzed images Critical
Extensibility Support custom detection plugins without core modification High
Testability Each detector must be independently testable High
Offline Operation No network dependencies during analysis Medium
Minimal Dependencies Self-contained, no external services required Medium

3. Detector Framework

The detector framework implements the Open/Closed Principle: open for extension, closed for modification. This is achieved through a trait-based plugin architecture.

3.1 Core Traits

// detector.rs

pub trait Detector: Send + Sync {
    fn name(&self) -> &'static str;
    fn detect(&self, ctx: &FileContext) -> Option<Finding>;
    fn severity(&self) -> Severity {
        Severity::Medium
    }
}

pub struct DetectorRegistry {
    detectors: Vec<Box<dyn Detector>>,
}

impl DetectorRegistry {
    pub fn register(&mut self, detector: Box<dyn Detector>);
    pub fn detect(&self, ctx: &FileContext) -> Vec<Finding>;
}

3.2 Detection Context

Each detector receives a FileContext containing pre-computed file attributes:

pub struct FileContext<'a> {
    pub path: &'a PathBuf,           // Absolute path
    pub relative_path: &'a PathBuf,   // Path relative to root
    pub mode: u32,                     // Unix permissions
    pub is_executable: bool,
    pub is_world_writable: bool,
    pub is_world_readable: bool,
    pub is_suid: bool,
    pub is_sgid: bool,
}

3.3 Finding Structure

pub struct Finding {
    pub kind: String,      // Detection type identifier
    pub path: PathBuf,      // File path
    pub detail: String,    // Human-readable details
    pub severity: Severity,// Critical/High/Medium/Low
}

pub enum Severity {
    Critical,  // Score: 5
    High,       // Score: 4
    Medium,     // Score: 3
    Low,        // Score: 2
}

4. Processing Pipeline

The analysis pipeline follows a strict sequence:

+------------------------+     +------------------------+
| Docker Image (.tar)   |---->| Image Loader           |
| Untrusted Input       |     | manifest.json parsing  |
+------------------------+     +-----------+------------+
                                        |
                                        v
                        +------------------------+
                        | Layer Verification    |
                        | locate_layers()       |
                        +-----------+------------+
                                        |
                                        v
                        +------------------------+
                        | Filesystem Build      |
                        | layer.tar extraction  |
                        +-----------+------------+
                                        |
                                        v
                        +------------------------+
                        | Scanner               |
                        | walk_dir()            |
                        +-----------+------------+
                                        |
                                        v
                        +------------------------+
                        | Detector Registry      |
                        | ctx = FileContext      |
                        | findings = detect(ctx)|
                        +-----------+------------+
                                        |
                                        v
                        +------------------------+
                        | Report Generator       |
                        | Filtering & Scoring   |
                        +------------------------+

4.1 Stage Details

Stage 1: Image Loading

Parses the Docker image tar archive using the tar crate. Extracts manifest.json to identify layer paths.

Stage 2: Layer Verification

Verifies all referenced layers exist in the tar archive. Returns error if any layer is missing.

Stage 3: Filesystem Reconstruction

Extracts each layer's layer.tar in order. Later layers override earlier ones. Handles whiteout files (markers for deleted files).

Stage 4: Directory Traversal

Recursively walks the reconstructed filesystem. For each file:

  • Collects file metadata (permissions, size, type)
  • Builds FileContext
  • Passes context to each registered detector
  • Collects all findings

Stage 5: Report Generation

Applies allowlist filtering, baseline comparison, and severity scoring. Outputs findings with suspicion level.

5. Trust Model

Security Assumption

The Docker image under analysis is treated as untrusted input. The analysis process must never execute any code from the image.

Component Trust Level Rationale
bsuppla Code Trusted Audited, version-controlled
Docker Image Untrusted External input, may be malicious
Analysis Output Trusted Generated by verified code

5.1 Security Boundaries

  • No Execution: Files are never opened with execute permissions
  • No Network: Analysis runs completely offline
  • Read-Only: Output directory is cleaned before each run
  • Isolated: Should run in containerized environment

6. Extensibility

Adding new detection capabilities requires implementing the Detector trait:

use crate::detector::{Detector, FileContext, Finding, Severity};

pub struct MyCustomDetector;

impl Detector for MyCustomDetector {
    fn name(&self) -> &'static str {
        "my_custom_detection"
    }

    fn detect(&self, ctx: &FileContext) -> Option<Finding> {
        // Custom detection logic
        if ctx.is_executable && /* ... */ {
            Some(Finding::new(
                "my_custom_detection",
                ctx.relative_path.clone(),
                "details here".to_string(),
                Severity::High,
            ))
        } else {
            None
        }
    }
}

Register the detector:

// In detectors/mod.rs
pub fn default_registry() -> DetectorRegistry {
    let mut registry = DetectorRegistry::new();
    // ... existing detectors ...
    registry.register(Box::new(my_custom::MyCustomDetector));
    registry
}

7. Conclusions

The bsuppla architecture provides:

  • Safe static analysis without code execution
  • Extensible plugin system via trait composition
  • Independent testability of each detector
  • Clear separation of concerns
